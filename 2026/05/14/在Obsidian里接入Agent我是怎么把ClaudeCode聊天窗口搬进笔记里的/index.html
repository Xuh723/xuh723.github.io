    在 Obsidian 里接入 Agent，我是怎么把 Claude Code 聊天窗口搬进笔记里的

    概述

    前几天看到一个 Obsidian Agent Client 插件，第一反应其实挺直接的：这不就是把 IDE 里的 agent 聊天体验搬进 Obsidian 吗。

    我一直有个很朴素的想法，既然写代码的时候已经习惯旁边挂一个 Claude Code、Codex 之类的 agent，那写笔记、整理知识库的时候，其实也应该有一个类似入口。不然一边在 Obsidian 里写，一边切回终端或者网页问 AI，来回跳还是有点碎。

    这篇文章就把这次折腾整理一下，主要讲三件事：

    • 这个插件到底解决了什么问题
    • 它是怎么接到 Claude Code CLI 上的
    • 如果自己想装，最短路径是什么

    这个插件到底是什么

    Obsidian Agent Client 是一个 Obsidian 社区插件，可以直接在 Obsidian 里接入 AI 代理。它不是只做一个”聊天框”，而是基于 ACP，也就是 Agent Client Protocol，去连接外部 agent。

    简单来说，它做的事是：

    • 在 Obsidian 里提供一个 agent 聊天入口
    • 把当前笔记环境、引用内容、对话历史这些东西接进来
    • 再把请求转给外部 agent，比如 Claude Code、Codex、Gemini CLI

    这点我觉得挺关键。因为它不是自己重新造一个 AI 客户端，而是把”Obsidian 作为工作台”和”本地 agent 作为执行能力”拼起来。

    对我来说，这个方向是对的。

    为什么我会想装这个东西

    平时如果只是临时问问题，终端里直接跑 claude 当然也够用。

    但放到知识库场景里，会多出几个很现实的需求：

    • 我想直接拿当前笔记当上下文
    • 我想在笔记里引用别的 note，而不是手动复制粘贴
    • 我想让 AI 帮我整理、改写、归纳内容之后，结果还能顺手落回 Obsidian

    如果这些动作还是全部绕回命令行，当然能做，只是入口不够顺。

    装上这个插件之后，感觉更像是在 Obsidian 里加了一个”侧边工作台”。写笔记的时候，agent 就在旁边，不需要总想着切出去。

    我在日记里记过一句很接近实际感受的话：这个插件挺符合我原本的预期，就像在 IDE 里面配一个 agent 聊天界面一样，只不过这次场景换成了 Obsidian。

    它能做什么

    从我整理的资料看，这个插件比较实用的点主要有这些：

    • 支持 @notename 引用笔记
    • 可以在多个 agent 之间切换
    • 支持多会话并行
    • 支持斜杠命令
    • 能把聊天内容导出成 Markdown 笔记
    • 带终端集成
    • 能复用 agent 自己已经配置好的 MCP

    这里面我最看重的其实不是”能聊天”，而是下面三点：

    1. 笔记能直接进入上下文

    这比复制粘贴省事很多。尤其是知识库里已经有不少历史记录时，引用已有笔记会自然很多。

    2. 不绑定单一模型

    它不是只能连一个 AI 提供商，而是通过 ACP 去接不同 agent。这意味着以后如果你换成别的 CLI，只要协议这层能通，整体工作流不用重搭。

    3. MCP 可以复用

    这点挺重要。因为很多真正好用的能力，最后都不是”纯聊天”，而是要连工具、连系统、连外部服务。插件如果能直接继承 Claude Code 那边已经配好的 MCP，省掉很多重复配置。

    主要看两个字段：

    一个我比较喜欢的点：可以不额外填 API key

    这也是我自己觉得比较符合预期的地方。

    如果本地 Claude Code CLI 已经登录好了，那么在插件里可以直接复用这套登录态，不一定非要再单独填一次 API key。

    这件事听起来只是少填一个字段，但体验上差别挺大。因为它意味着：

    • CLI 和 Obsidian 不是两套完全割裂的系统
    • 你已经配置好的 agent 环境可以直接沿用
    • 后面如果 CLI 里还有 MCP、工作流习惯、工具能力，也更容易一起复用

    换句话说，它不是重新造一个 AI 入口，而是把你现有的入口接进笔记里。

    这个插件更适合什么人

    我觉得它特别适合下面这类使用方式：

    1. 已经在用 Claude Code / Codex 这类 CLI 的人

    如果你平时本来就习惯在终端里用 agent，这个插件会比较自然。因为你不是从零学一套新东西，而是多了一个更顺手的图形入口。

    2. Obsidian 已经是主工作台的人

    如果你写笔记、做知识管理、记工程备忘、整理项目材料都在 Obsidian 里，那把 agent 放进来很有意义。

    3. 想把 AI 用在”知识工作流”里的人

    不是只想问几个问题，而是想做下面这些事：

    • 改写笔记
    • 总结已有内容
    • 对比多篇记录
    • 从历史知识里提炼结构
    • 把结果重新沉淀成文档

    这种场景里，agent 离笔记越近，体验越顺。

    它的边界也很明显

    当然，这个插件也不是装上之后世界就清净了。

    我现在的感觉是，它更像一个”好入口”，不是完整答案。

    比如：

    • 真正决定效果的，还是你本地 agent 怎么配
    • Obsidian 里有了聊天框，不等于知识库工作流就自动跑通了
    • 如果你本来就能熟练在命令行里操作 Obsidian，这个插件带来的更多是便利，不是能力从无到有

    所以别把它理解成”装完就拥有第二大脑”，那会容易期待过高。

    更准确一点说，它是把 AI 接近你的知识库工作台，但后面能不能真正提效，还是得看你怎么设计自己的使用方式。

    我现在对它的判断

    我还是挺看好这类插件的。

    因为它代表了一个我一直觉得重要的方向：AI 不应该只存在于网页标签页里，而应该慢慢接进你真正工作的地方。

    对写代码的人来说，这个地方可能是 IDE。

    对长期做知识积累的人来说，这个地方很可能就是 Obsidian。

    把 agent 接到这里之后，最大的价值不是”多了个聊天窗口”，而是：

    • 上下文离你的资料更近了
    • 处理动作离你的文档更近了
    • 知识沉淀和 AI 使用之间的距离更短了

    这一步看起来不大，但方向是对的。
